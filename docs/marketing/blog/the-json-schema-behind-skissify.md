# The JSON Schema Behind Skissify — How We Made AI Agents into Designers

*~2,000 words | 9 min read*

---

When I decided AI agents should be able to draw, I ran into a problem no one had solved cleanly: what's the right data format for a sketch?

This isn't an obvious question. Every existing drawing tool has its own format — Excalidraw's `.excalidraw` JSON, Mermaid's text DSL, SVG, Canvas instructions — and none of them are designed to be generated by an AI agent from natural language.

Here's what I learned designing Skissify's schema, and why those choices shaped the entire product.

---

## The Core Constraint: LLMs Must Be Able to Write It

The first requirement was brutal: **an LLM with no training on Skissify should be able to write valid JSON on the first try, from a natural language description.**

That rules out a lot of approaches:

- **SVG**: Too verbose, too many attributes, too much knowledge of XML namespaces required
- **Excalidraw format**: Coordinates are viewport-relative and require knowledge of previous elements to position new ones
- **Canvas API calls**: Imperative, stateful, requires understanding render order semantics
- **Mermaid/Graphviz DSL**: Text-based, not spatial, requires specific syntax knowledge

What LLMs are good at: straightforward JSON with clear, named properties. What they're bad at: inferring implicit state, managing coordinate systems that accumulate transformations, or handling DSL syntax they haven't seen.

The schema needed to be **obvious**. A dev who's never seen Skissify should look at the JSON and understand it instantly.

## The Schema Design

Here's the current Skissify sketch manifest:

```json
{
  "paper": "cream",
  "tool": "ballpoint",
  "amplitude": 3,
  "waves": 4,
  "humanness": 0.4,
  "width": 800,
  "height": 600,
  "elements": [
    { "type": "rect", "x": 50, "y": 50, "w": 400, "h": 300, "label": "Living Room" },
    { "type": "window", "x": 150, "y": 50, "w": 100, "h": 20 },
    { "type": "door-symbol", "x": 400, "y": 175, "w": 80, "h": 20, "angle": 270 },
    { "type": "dim", "x": 50, "y": 380, "x2": 450, "y2": 380, "label": "8.0m" },
    { "type": "text", "x": 200, "y": 200, "label": "North ↑" }
  ]
}
```

### Top-Level Properties

The top level handles aesthetics and canvas setup. Every property is optional with sensible defaults:

| Property | Type | Default | Description |
|----------|------|---------|-------------|
| `paper` | string | `"cream"` | Paper texture: `cream`, `white`, `grid`, `blueprint` |
| `tool` | string | `"ballpoint"` | Drawing tool: `pencil`, `ballpoint`, `ink` |
| `amplitude` | number | `3` | Wobble size (0 = straight, 10 = drunk) |
| `waves` | number | `4` | Wobble frequency |
| `humanness` | number | `0.4` | Random tremor overlay (0–1) |
| `width` | number | `800` | Canvas width in pixels |
| `height` | number | `600` | Canvas height in pixels |

This is important: **top-level aesthetic properties apply globally**. The AI doesn't need to set style on every element — it sets it once and all elements inherit.

### Element Properties

Every element has a `type` and position. The position model is **absolute coordinates** from the top-left:

```json
{ "type": "rect", "x": 50, "y": 50, "w": 400, "h": 300 }
```

The deliberate choice here: **no relative positioning, no transforms, no parent-child relationships**. An element's position is always `x, y` from canvas origin.

This is slightly verbose (you can't say "put this room next to that room"), but it's trivially generatable by an LLM. When Claude designs a floor plan, it figures out the spatial layout itself and writes absolute coordinates. This maps exactly to how humans think: "the bedroom starts at 300, 200 and is 200 wide by 150 tall."

## The 14 Element Types

The element type system covers most architectural floor plan needs. Here's the taxonomy:

### Geometric Primitives
```json
{ "type": "line", "x": 0, "y": 0, "x2": 100, "y2": 0 }
{ "type": "rect", "x": 50, "y": 50, "w": 200, "h": 150, "label": "Kitchen" }
{ "type": "circle", "x": 200, "y": 200, "r": 50 }
{ "type": "arc", "x": 100, "y": 100, "r": 80, "startAngle": 0, "endAngle": 90 }
{ "type": "arrow", "x": 100, "y": 100, "x2": 200, "y2": 100, "label": "traffic flow" }
{ "type": "dashed", "x": 0, "y": 0, "x2": 100, "y2": 100 }
```

### Annotation Types
```json
{ "type": "text", "x": 100, "y": 100, "label": "North ↑", "fontSize": 14 }
{ "type": "dim", "x": 50, "y": 380, "x2": 450, "y2": 380, "label": "12.5m" }
```

The `dim` type is worth calling out: it renders as a dimension line with end ticks and a measurement label, centered above the line. Architects use these constantly. Having it as a named type instead of manually composing it from lines + text was a big usability win.

### Architectural Specifics
```json
{ "type": "window", "x": 150, "y": 50, "w": 100, "h": 20 }
{ "type": "door-symbol", "x": 400, "y": 200, "w": 80, "h": 20, "angle": 90 }
{ "type": "door-slide", "x": 200, "y": 300, "w": 80, "h": 20 }
{ "type": "opening", "x": 100, "y": 300, "w": 60, "h": 20 }
{ "type": "stair", "x": 300, "y": 200, "w": 80, "h": 120, "steps": 8 }
{ "type": "column", "x": 200, "y": 200, "r": 15 }
```

The `door-symbol` element is the one that most impresses people. It renders the classic hinged door symbol: a small square (the door) with an arc sweeping out to show the swing radius. You specify `angle` to indicate which direction the door swings. AI agents pick this up immediately — Claude consistently generates valid door-symbols without instruction.

## The Wobble Engine

The aesthetic half of Skissify is the wobble algorithm. This is worth detailing because it's non-obvious.

### Why Not a Simple Sine Wave?

Every "hand-drawn" style tool I tried before building Skissify used the same approach: for each point on a line, offset it slightly by `sin(x * frequency) * amplitude`. 

The result looks mechanical. Human hands don't oscillate at a constant frequency. The wobble is too regular, too predictable.

### Multi-Harmonic Wobble

Skissify's approach uses multiple overlapping sine waves at different frequencies, combined with a seeded random tremor:

```typescript
function wobbledPoint(x: number, y: number, elementSeed: number, params: WobbleParams): Point {
  const t = Math.hypot(x, y); // distance from origin as phase input
  
  // Primary harmonic
  const h1 = Math.sin(t * params.waves * 0.01) * params.amplitude;
  // Second harmonic (different frequency)
  const h2 = Math.sin(t * params.waves * 0.027 + elementSeed) * (params.amplitude * 0.4);
  // Third harmonic (high frequency, low amplitude)
  const h3 = Math.sin(t * params.waves * 0.08 + elementSeed * 2) * (params.amplitude * 0.15);
  
  // Random tremor (humanness)
  const tremor = (seededRandom(elementSeed + t) - 0.5) * params.humanness * params.amplitude;
  
  const offset = h1 + h2 + h3 + tremor;
  
  // Apply perpendicular to line direction
  return applyPerpendicularOffset(x, y, offset);
}
```

The `elementSeed` is derived from each element's position in the manifest, so every element has unique wobble even with identical parameters. Wall at `x:100` looks different from wall at `x:200`.

### Per-Element Wobble Tuning

Each element type has tuned wobble behavior applied before the global parameters:

- **Dimension lines** (`dim`): tight tolerances, low amplitude multiplier — they should look precise
- **Door arcs**: medium wobble, different phase offset — the curved arc gets unique treatment
- **Structural columns** (`column`): low wobble — these are meant to look solid
- **Freeform text**: high humanness multiplier — handwriting looks less regular
- **Stairs** (`stair`): the individual treads get wobbled at a different phase than the enclosing rectangle

This is the detail that makes the output feel genuinely hand-drawn rather than algorithmically generated.

## How AI Agents Use the Schema

With the JSON schema in place, getting AI agents to use it correctly was surprisingly straightforward.

The system prompt we give Claude via the MCP tool description:

```
create_sketch: Renders a Skissify JSON manifest as a hand-drawn sketch.

Input: JSON object with:
- paper: "cream" | "white" | "grid" | "blueprint" (default: "cream")
- tool: "pencil" | "ballpoint" | "ink" (default: "ballpoint")
- amplitude: 0-10 (line wobble, default: 3)
- waves: 1-10 (wobble frequency, default: 4)
- humanness: 0-1 (random tremor, default: 0.4)
- elements: array of element objects

Element types: rect, line, circle, arc, arrow, dashed, text, dim, window, 
door-symbol, door-slide, opening, stair, column

Each element: { type, x, y, w?, h?, x2?, y2?, r?, label?, angle?, steps? }
Coordinates are absolute pixels from top-left.

Returns: { url: "https://skissify.com/s/[id]", id: "[id]" }
```

That description is enough. Claude generates valid JSON on the first attempt about 85% of the time in testing. The 15% failure rate is almost always a missing `w`/`h` on a `rect` or a slightly out-of-range `humanness` value — trivial validation catches these.

## What I Got Wrong First

Three schema mistakes I made and fixed:

**1. Relative coordinates ("next to room A")**  
I tried a relative positioning system (`after: "room-a"`, `offset: {x: 10, y: 0}`). LLMs found it confusing and generated broken layouts. Absolute coordinates are more verbose but unambiguous.

**2. Style properties on individual elements**  
I tried `{ "type": "rect", "stroke": "#333", "fill": "transparent", "strokeWidth": 2 }`. Immediately problematic: LLMs would set conflicting styles, break the hand-drawn aesthetic, and produce inconsistent outputs. Moving all aesthetic control to the top level was a massive improvement.

**3. Named elements with IDs for referencing**  
I tried `{ "id": "bedroom-1", "type": "rect", ... }` so elements could reference each other. Never worked cleanly. Dropped IDs entirely — the JSON is a flat list, and positions encode relationships.

## What's Next for the Schema

Current gaps and planned additions:

- **Furniture blocks**: sofa, bed, desk, table (with standard architectural symbols)
- **Electrical symbols**: outlet, switch, light fixture
- **Plumbing**: sink, toilet, shower
- **Rotation**: `angle` property on `rect` and `line` (currently only `door-symbol` uses it)
- **Groups**: logical grouping for selection and animation (useful for interactive editor)
- **SVG export**: the Canvas 2D output needs an SVG path for this, in progress

The schema will remain flat and LLM-generatable. That constraint is the product.
