# How AI Agents Draw Floor Plans (The Incident Architecture Diagram Problem)

*Posted: April 4, 2026 — Cycle 138*

---

Production goes down at 2 AM. You're in a war room. Everyone is shouting names of services.

You know what nobody has? A current architecture diagram.

---

## The Diagram That's Always Out of Date

Every engineering team has a system architecture diagram. It lives in Confluence or Notion or Google Drive. It was accurate the day someone drew it — usually during the initial design phase — and has been slowly drifting from reality ever since.

When incidents happen, engineers pull up the diagram. It's missing three services added in Q3. It doesn't show the new message queue. The database it shows was migrated six months ago.

The diagram isn't wrong because nobody cares. It's wrong because updating diagrams is slow and the reward is invisible until something breaks.

---

## What Changes When the Agent Draws the Diagram

An AI agent connected to Skissify can generate an architecture diagram from your runbook, your service registry, or even a plain-English description of the current state.

```json
{
  "paper": "blueprint",
  "tool": "ink",
  "elements": [
    { "type": "rect", "id": "api-gateway", "label": "API Gateway", ... },
    { "type": "rect", "id": "auth-service", "label": "Auth Service", ... },
    { "type": "arrow", "from": "api-gateway", "to": "auth-service" },
    ...
  ]
}
```

The agent doesn't draw the diagram once. It can regenerate it from current state any time you ask. The diagram becomes a query, not a document.

---

## Three Incident Use Cases

**1. War room context sync**
When an incident starts, run: "Generate the architecture diagram for the payment service cluster." Everyone joins the call with the same picture in their browser.

**2. Post-incident timeline reconstruction**
"Draw a sequence diagram of what we know happened between 2:14 AM and 2:38 AM based on this incident log." The sketch externalizes the investigation.

**3. Runbook visual aids**
Every runbook step that says "see the diagram in Confluence" can instead say "generate the current diagram with: [prompt]." The diagram is always fresh.

---

## Beyond Incidents: The Living Diagram Pattern

The same pattern applies anywhere diagrams go stale:

- **Onboarding docs** — generate the current service map when a new engineer joins
- **Sprint planning** — sketch the proposed architecture before committing to it
- **Customer demos** — generate a simplified diagram showing only the relevant parts of the system
- **Code review** — attach a sketch URL to PRs that change service relationships

The sketch doesn't replace the code. It makes the code visible to everyone in the room.

## Getting Started

Three setup options:

**1. Claude MCP (fastest)**
```bash
npx -y @skissify/mcp-server
```
Then in Claude: "Use skissify to draw the architecture for [describe your system]."

**2. REST API**
```bash
curl -X POST https://api.skissify.com/render \
  -H "Content-Type: application/json" \
  -d '{ "paper": "blueprint", "elements": [...] }'
```

**3. Human mode (no code)**
[skissify.com/human](https://skissify.com/human) — describe your system in plain English, get a sketch URL.

## The Deeper Point

Architecture diagrams are expensive because they require a human to maintain them. When a diagram is generated by an agent from current state, the cost disappears. The diagram is always as fresh as the last time someone asked for it.

That's the real shift: from diagrams as documents to diagrams as outputs.
