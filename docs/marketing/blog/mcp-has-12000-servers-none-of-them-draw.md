PulseMCP lists 12,770 Model Context Protocol servers as of this week.

12,770.

There are MCP servers for Slack, Notion, GitHub, Jira, Stripe, Postgres, weather, email, stock prices, web scraping, SSH, Redis, MongoDB, Kubernetes, and roughly 12,760 other things.

But until today, there wasn't one that could draw.

## What's Actually Missing

When Anthropic released MCP in late 2024, it was designed to answer a simple question: *how do AI agents interact with the world?*

The answer: structured tool calls. The AI tells the server what to do; the server does it; the AI incorporates the result into its reasoning.

That works brilliantly for data retrieval. Fetch the Jira issue. Query the database. Get the current temperature in Stockholm.

But there's a class of AI tasks that aren't about retrieval — they're about **creation**. And specifically, *visual* creation.

AI language models can reason spatially. Ask Claude to "design a compact home office for a developer with two monitors and a standing desk option" and you'll get a thoughtful, well-considered layout. Room dimensions. Furniture placement. Traffic flow.

The reasoning is already there. What wasn't there: a way to output it visually.

## Why Not Just Use Image Generation?

Fair question. DALL-E and Midjourney and Stable Diffusion exist. Why not just describe the floor plan and let an image model render it?

Three problems:

**1. Precision.** Image generators are stochastic. The same prompt generates different outputs every time. For floor plans, architecture diagrams, or wireframes, you need determinism: the same JSON input always produces the same visual output.

**2. Correctness.** Image generators frequently get spatial reasoning wrong. Rooms bleed into each other. Doors appear in wrong places. Dimensions are fictional. A floor plan generated by DALL-E is aesthetically plausible but architecturally nonsensical.

**3. Programmatic control.** You can't ask an image generator "make the living room 20% larger" and have it do that precisely. You can ask Skissify — you just change the `w` parameter.

What's actually needed for AI agents isn't an image generator. It's a renderer: a system that takes structured data and produces consistent, correct visual output.

## The JSON-First Approach

This is the core design decision behind Skissify.

Every sketch is a JSON manifest. No drag-and-drop. No canvas. No mouse.

```json
{
  "paper": "cream",
  "tool": "ballpoint",
  "amplitude": 3,
  "humanness": 0.4,
  "elements": [
    { "type": "rect", "x": 50, "y": 50, "w": 350, "h": 220, "label": "Living Room" },
    { "type": "rect", "x": 50, "y": 270, "w": 160, "h": 140, "label": "Kitchen" },
    { "type": "rect", "x": 240, "y": 270, "w": 160, "h": 140, "label": "Bedroom" },
    { "type": "door-symbol", "x": 165, "y": 50, "w": 70, "h": 20 },
    { "type": "window", "x": 50, "y": 140, "w": 90, "h": 20 },
    { "type": "dim", "x": 50, "y": 430, "x2": 400, "y2": 430, "label": "12.5m" }
  ]
}
```

→ Hand-drawn floor plan. Every time.

The JSON-first constraint isn't a limitation — it's what makes the tool useful to agents. Anything that can generate JSON can generate Skissify sketches. That's a much larger set than "anything that can operate a mouse."

## The MCP Integration

The Skissify MCP server exposes one primary tool: `create_sketch`.

Call it with a JSON manifest. Get back a rendered sketch URL. That's it.

Within Claude Desktop, the workflow looks like this:

> **User:** Design a small studio apartment for a remote worker. Show me the floor plan.
>
> **Claude:** I'll design this with the work area as the priority — good natural light and acoustic separation from the sleeping area...
>
> *[calls create_sketch with generated JSON]*
>
> Here's the floor plan: [rendered image]
>
> The home office corner (3.2m × 2.8m) is at the east wall to capture morning light...

The AI designed the space, generated the JSON, called Skissify, and returned a sketch — all in a single turn. No human hand in the loop.

## Why This Timing

Three forces converged in 2026 that made this tool viable:

**MCP critical mass.** With 12,000+ servers and Claude, Continue, Zed, and other clients shipping native MCP support, the distribution problem is solved. Building an MCP server in 2024 meant explaining what MCP was to every user. Building one in 2026 means they already have it set up.

**LLM spatial reasoning improved.** The success rate for LLM-generated Skissify JSON is around 88-92% for floor plans (measured on 200+ test generations). That's high enough to be actually useful. In 2024, it would have been in the 40-60% range — not practical.

**Developer appetite for agent-native tools.** The market has moved from "AI-assisted tools" (human in the loop, AI as autocomplete) to "agent-native tools" (AI can operate the tool without a human present). Skissify was designed from day one for the second category.

## What "Agent-Native" Actually Means in Practice

It means different things for different tool categories. For Skissify, it specifically means:

**Determinism over creativity.** The same input always produces the same output. Agents can reason about what they'll get.

**Structured input.** No natural language parsing needed. The agent generates JSON; the tool renders it. Simple contract.

**Error tolerance.** If an element has malformed coordinates, the renderer skips it gracefully rather than crashing. Agents make mistakes; the tool should degrade well.

**API-level speed.** No browser automation, no screenshot capture, no waiting. Render happens server-side in <100ms. Agents can make multiple calls in a single conversation turn.

**Programmatic aesthetic control.** The agent can specify how hand-drawn the output should look. Need a precise technical drawing? `humanness: 0.1`. Need a quick napkin sketch? `humanness: 0.9`. The aesthetic is a parameter, not a fixed style.

## The 12,770 Are Tools. Skissify Is Output.

Here's the framing that clarifies everything:

Most MCP servers are **input tools**: they give the AI access to data, state, or actions in external systems. The AI reads from them, writes to them, manipulates them.

Skissify is an **output tool**: it turns the AI's reasoning into a visual artifact. The AI produces something; Skissify renders it.

That's a different job than 95% of what's in the MCP directory. And it's the job that's been missing.

## Try It

For Claude Desktop integration:
```bash
npm install -g @skissify/mcp-server
```

Then add to your `claude_desktop_config.json`:
```json
{
  "mcpServers": {

      "command": "npx",
      "args": ["@skissify/mcp-server"],
      "env": { "SKISSIFY_API_KEY": "your-api-key" }
    }
  }
}
```

Ask Claude to "draw me a floor plan" and see what happens.
